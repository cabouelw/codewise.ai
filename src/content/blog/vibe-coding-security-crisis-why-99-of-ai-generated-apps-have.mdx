The term "vibe coding" has exploded across developer Twitter, Reddit, and Discord servers over the past year. It describes a workflow where developers—sometimes experienced, often not—prompt an LLM to generate entire features, authentication flows, database schemas, and deployment configs, then ship the output with minimal review. The vibe is fast. The vibe is fun. The vibe is also dangerous.

Recent analyses from security firms including Synk, GitGuardian, and independent researchers converge on a staggering statistic: **approximately 99% of AI-generated codebases contain at least one critical vulnerability**. That's not a typo. Ninety-nine percent.

If you've copied a `supabase` auth helper from ChatGPT, asked Cursor to "build me a Stripe checkout flow," or had Copilot scaffold a REST API with "proper error handling," this post is for you. Let's break down why this happens, what the most common vulnerabilities look like, and how to keep shipping fast without shipping exploits.

---

## The Root Cause: Context-Free Generation

LLMs don't understand your architecture. They don't know your threat model. They don't know that your `/api/users/:id` endpoint sits behind an API gateway that *should* handle auth but currently doesn't. They predict tokens based on training data—and that training data is full of insecure patterns.

### The "Tutorial Code" Problem

Most code in an LLM's training corpus comes from tutorials, Stack Overflow answers, and open-source repos that prioritize *demonstrating a concept* over *production hardening*. When you ask "show me a Node.js login endpoint," the model retrieves patterns like:

```javascript
// ❌ Vibe-coded: looks fine, ships broken
app.post('/login', async (req, res) => {
  const { email, password } = req.body;
  const user = await db.users.findOne({ email });
  
  if (user && await bcrypt.compare(password, user.passwordHash)) {
    const token = jwt.sign({ userId: user.id }, 'secret123'); // Hardcoded secret!
    return res.json({ token });
  }
  
  res.status(401).json({ error: 'Invalid credentials' });
});
```

This code *works*. It also:
- Uses a hardcoded JWT secret
- Lacks rate limiting (credential stuffing welcome)
- Returns the same error for "user not found" and "wrong password" (user enumeration)
- No secure cookie flags, no CSRF protection, no audit logging

The model didn't *choose* insecurity. It just reproduced the most statistically probable answer for "login endpoint."

## The Top 5 Vulnerabilities in Vibe-Coded Apps

### 1. Broken Access Control (IDOR Everywhere)

AI loves generating resource endpoints with direct object references and zero authorization checks.

```python
# ❌ Generated by "create a REST API for user profiles"
@app.get("/api/users/{user_id}/profile")
async def get_profile(user_id: str):
    return await db.profiles.find_one({"user_id": user_id})

# Attacker hits: GET /api/users/1337/profile → gets PII
```

**Fix:** Every resource access needs an authorization decision. Use a policy engine (Oso, Casbin) or at minimum a middleware that verifies `request.user.id === resource.owner_id`.

### 2. Injection Flaws Beyond SQL

SQL injection gets attention. But AI-generated code introduces injection in GraphQL resolvers, ORM raw queries, template engines, and shell commands.

```typescript
// ❌ "Build me a search endpoint"
app.get('/search', async (req, res) => {
  const results = await prisma.$queryRaw`
    SELECT * FROM products WHERE name ILIKE '%${req.query.q}%'
  `;
  res.json(results);
});

// Payload: ' UNION SELECT password FROM users--
```

**Fix:** Never interpolate user input into raw queries. Use parameterized queries, ORM safe methods, or validated allow-lists.

### 3. Secrets in Plain Sight

LLMs frequently hardcode API keys, database URLs, and encryption keys because tutorials do it.

```python
# ❌ Actual Copilot suggestion for "Stripe webhook handler"
stripe.api_key = "sk_live_51H..."  # Real key in training data!

# Or environment variable fallbacks that leak in logs
DATABASE_URL = os.getenv("DATABASE_URL", "postgres://user:pass@localhost/db")
```

**Fix:** Use a secrets manager (Doppler, 1Password CLI, AWS Secrets Manager). Never default secrets. Fail loudly if missing.

### 4. Insecure Defaults in Auth Flows

AI generates auth code that *looks* complete but misses critical settings:

```javascript
// ❌ Missing: secure, httpOnly, sameSite, domain, path
res.cookie('session', token, { maxAge: 86400000 });

// ❌ No PKCE, no state parameter, redirect URI not validated
app.get('/auth/callback', async (req, res) => {
  const tokens = await exchangeCode(req.query.code);
  res.redirect(`/dashboard?token=${tokens.access_token}`); // Token in URL!
});
```

**Fix:** Use battle-tested libraries (Auth.js, Lucia, Clerk). Don't roll your own OAuth.

### 5. Supply Chain & Dependency Confusion

Vibe coding often means "add this npm package" without auditing. AI suggests popular but unmaintained packages, or hallucinates package names that attackers have already registered (dependency confusion / typosquatting).

## A Practical Security Review Checklist for AI-Generated Code

Before merging *any* AI-assisted PR, run through this 10-minute checklist:

| Check | How to Verify |
|-------|---------------|
| **No hardcoded secrets** | `git log --all --full-history --oneline --source --remotes --grep="secret\|key\|token\|password" -- "*"` + `trufflehog` / `gitguardian` |
| **AuthZ on every endpoint** | Search for route handlers missing `requireAuth` or `requireOwnership` middleware |
| **Input validation** | Every `req.body`, `req.query`, `req.params` passes through Zod / Joi / Pydantic schema |
| **Rate limiting** | Auth endpoints: ≤5 req/min. API: ≤100 req/min. Use Redis-backed limiter |
| **Secure headers** | `helmet()` (Express), `secure-headers` (Rails), `django-secure` — verify CSP, HSTS, X-Frame-Options |
| **Error handling** | No stack traces in production responses. Generic messages: "Internal server error" |
| **Dependency audit** | `npm audit --audit-level=high` / `pip-audit` / `cargo audit` — fail CI on critical |
| **Logging & monitoring** | Auth failures, privilege escalations, and validation errors emit structured logs |

## Tooling That Actually Helps

You don't need to manual-review every line. Layer these into your pipeline:

### Static Analysis (Run in CI)
```yaml
# .github/workflows/security.yml
- name: Run Semgrep
  uses: returntocorp/semgrep-action@v1
  with:
    config: >-
      p/security-audit
      p/secrets
      p/owasp-top-ten
```

**Semgrep rules** catch the patterns LLMs repeat: hardcoded secrets, missing auth checks, unsafe deserialization, path traversal.

### Runtime Protection
- **Arcjet** or **Cloudflare WAF** for bot detection, rate limiting, attack signatures at edge
- **Sentry** + custom alerts for auth anomalies (10 failed logins from same IP in 1 minute)

### Dependency Health
```bash
# Add to package.json scripts
"security:deps": "npm audit --audit-level=high && npx depcheck"
```

### AI-Specific Guardrails
- **GitHub Copilot Chat**: `/explain` + "what security issues exist in this code?"
- **Cursor**: `@codebase` "find all endpoints missing authorization middleware"
- **Local LLMs** (Ollama + CodeLlama): Run security-focused prompts offline
