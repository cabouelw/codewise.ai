If you've ever watched a browser freeze while parsing a massive JSON file or processing a large CSV, you know the pain. JavaScript is single-threaded, garbage-collected, and—let's be honest—not designed for heavy computational lifting. But what if you could keep your JavaScript UI responsive while offloading the parsing grunt work to a module running at near-native speed?

That's exactly what WebAssembly (Wasm) delivers. In this post, I'll walk you through how we achieved a **350% performance improvement** on a production text parser by moving the hot path to WebAssembly, and I'll show you the patterns you can apply to your own projects today.

---

## Why Text Parsing Is a Perfect Wasm Candidate

Text parsing hits every JavaScript weakness simultaneously:

- **String manipulation overhead**: JS strings are UTF-16 immutable sequences. Every `slice`, `split`, or regex match allocates new memory.
- **Garbage collection pressure**: Parsing generates thousands of short-lived objects—tokens, AST nodes, substring copies. The GC pauses become visible as jank.
- **No SIMD, no manual memory control**: You can't tell V8 to "just use this pre-allocated buffer" or "process 16 bytes at a time with AVX2."

WebAssembly flips this. You get:
- Linear memory you control
- SIMD instructions (via `wasm_simd128` proposal, now widely supported)
- Zero GC pauses for the parsing logic itself
- Predictable, measurable performance

## The Baseline: A JavaScript CSV Parser

Let's start with a realistic scenario. We need to parse a 50 MB CSV file—about 1.2 million rows, 15 columns each. Here's a simplified version of what many teams ship:

```javascript
function parseCSV(text) {
  const lines = text.split('\n');
  const headers = lines[0].split(',');
  const rows = [];
  
  for (let i = 1; i < lines.length; i++) {
    const values = lines[i].split(',');
    if (values.length !== headers.length) continue;
    
    const row = {};
    for (let j = 0; j < headers.length; j++) {
      row[headers[j]] = values[j];
    }
    rows.push(row);
  }
  return rows;
}
```

**Performance on a 2023 M2 MacBook Pro (Chrome 119):**
| Metric | Value |
|--------|-------|
| Parse time | ~2,840 ms |
| Peak memory | ~420 MB |
| Main thread blocked | ~2.8 s |

That's a frozen UI for nearly 3 seconds. Unacceptable for a responsive app.

## Step 1: Design the Wasm Interface

The golden rule: **minimize boundary crossings**. Every call from JS → Wasm → JS costs ~50-100ns. Don't call Wasm per token. Pass the whole buffer, get back a structured result.

We'll use Rust with `wasm-bindgen`—but the same principles apply to C++, AssemblyScript, or Zig.

```rust
// lib.rs
use wasm_bindgen::prelude::*;

#[wasm_bindgen]
pub fn parse_csv(input: &str) -> JsValue {
    // Parsing logic here
    // Return a JsValue (array of objects) via serde_wasm_bindgen
}
```

**Key insight**: Accept `&str` (UTF-8) from JS. Wasm memory is linear; `wasm-bindgen` handles the UTF-16 → UTF-8 copy once, upfront. That's one allocation, not millions.

## Step 2: Implement a Streaming, Zero-Copy Parser

Here's where the magic happens. Instead of splitting into arrays, we iterate bytes once, track column boundaries, and write directly into a pre-allocated output buffer.

```rust
pub fn parse_csv(input: &str) -> Vec<Vec<&str>> {
    let bytes = input.as_bytes();
    let mut rows = Vec::with_capacity(estimate_rows(bytes));
    let mut row = Vec::with_capacity(15);
    let mut col_start = 0;
    
    for (i, &b) in bytes.iter().enumerate() {
        match b {
            b',' => {
                row.push(unsafe { std::str::from_utf8_unchecked(&bytes[col_start..i]) });
                col_start = i + 1;
            }
            b'\n' => {
                row.push(unsafe { std::str::from_utf8_unchecked(&bytes[col_start..i]) });
                rows.push(std::mem::take(&mut row));
                row = Vec::with_capacity(15);
                col_start = i + 1;
            }
            _ => {}
        }
    }
    rows
}
```

**Why this screams:**
- **Zero allocations per field**: We return `&str` slices into the original input buffer
- **Single pass**: One loop, branch-predictor-friendly
- **Pre-sized vectors**: `with_capacity` eliminates reallocations
- **Unsafe only where proven safe**: Bounds checked by loop logic

## Step 3: Add SIMD for the Hot Loop

Modern browsers support `wasm_simd128`. We can scan 16 bytes at a time for delimiters:

```rust
#[cfg(target_feature = "simd128")]
fn find_delimiters_simd(bytes: &[u8]) -> Vec<usize> {
    use std::simd::{u8x16, SimdPartialEq};
    
    let comma = u8x16::splat(b',');
    let newline = u8x16::splat(b'\n');
    let mut positions = Vec::new();
    
    for (chunk_idx, chunk) in bytes.chunks_exact(16).enumerate() {
        let v = u8x16::from_slice(chunk);
        let mask = v.simd_eq(comma) | v.simd_eq(newline);
        let bitmask = mask.to_bitmask();
        
        if bitmask != 0 {
            for bit in 0..16 {
                if bitmask & (1 << bit) != 0 {
                    positions.push(chunk_idx * 16 + bit);
                }
            }
        }
    }
    positions
}
```

**Result**: The delimiter scan runs ~4x faster on AVX2-capable hardware. Fallback to scalar for older CPUs—`#[cfg]` handles this cleanly.

## Step 4: Benchmark Honestly

Always measure with realistic data, not microbenchmarks. Our test harness:

```javascript
// benchmark.js
const fs = require('fs');
const { parse_csv } = require('./pkg/wasm_parser.js');

const csvData = fs.readFileSync('realistic_50mb.csv', 'utf-8');

// Warmup
for (let i = 0; i < 3; i++) parse_csv(csvData.slice(0, 10000));

// Measured runs
const times = [];
for (let i = 0; i < 10; i++) {
  const start = performance.now();
  parse_csv(csvData);
  times.push(performance.now() - start);
}

console.log('Median:', median(times), 'ms');
```

**Results after optimization:**

| Implementation | Median Time | Speedup | Peak Memory |
|----------------|-------------|---------|-------------|
| JS (baseline) | 2,840 ms | 1.0x | 420 MB |
| Wasm (scalar) | 1,120 ms | 2.5x | 180 MB |
| Wasm + SIMD | **790 ms** | **3.6x** | **95 MB** |

That's **350% faster** (3.6x) with **77% less memory**. The main thread stays free—parsing happens in a Web Worker, posting back the result.
